#각 팀별 가이드라인

하나의 프로젝트에서도 다양한 역할이 필요합니다. 기획, 디자인, 개발 등 다양한 직군과 함께 프로젝트를 진행할 수 있습니다. 물론, 개인 프로젝트의 경우 해당 역할에 배정된 할 일들을 스스로 해야합니다. 이번 챕터에서는 하나의 도구를 선택하여 설명하기 보단 대부분의 도구들을 사용할 수 있는 일반적인 방법에 대해서 알려드리려고 합니다.

### 04.1.1 컨벤션을 정해야 하는 이유

**- 컨벤션이란**

 소프트웨어 개발 분야에서 "컨벤션"이란 코드 작성, 프로젝트 구성, 명명 규칙 등 다양한 측면에서 관행, 규칙을 정의하는 것을 말합니다. 게시글 페이지를 만든다고 한다면, 해당 페이지를 `feed`라고 부를지 `post`라고 부를지 등을 정하는 것입니다. 주로 개발자들이 정하는 것이라고 생각할 수 있으나, 컨벤션은 대부분 모든 직군들이 참여하여 정하는 내용이 있고, 해당 직군 별로 작업을 할 때 쓰기 위해 정하는 내용이 있습니다.


**- 컨벤션을 정해야 하는 이유**

앞에서 말했던 내용에 덧붙이자면, PM은 `feed`라 부르는 게시판 페이지를 디자이너는 `post`라고 부르고 있다면, 서로 소통하는데 많은 비용이 듭니다.

데이터를 다룰 때도 마찬가지로, PM이 말하는 `feed`와 디자이너가 말하는 `post`를 별개의 페이지로 듣고 있던 개발자가 있다고 생각해봅시다. 그리고 개발자들은 `feed`와 `post`를 위해 서로 역할을 분담하여 한 명은 `feed`를, 다른 한 명은 `post`에 들어갈 데이터를 작성해오라고 했습니다.
```
feed{
 ID: 1234
}

post{
 post_ID: 1234
}
```
위처럼 같은 ID라는 데이터에 접근하기 위해서는 각각의 데이터를 모두 잘 알고 있어야 합니다. 또한, 이름이 다르기 때문에 같은 일을 하는 로직을 두 번 즉, `post_ID`와 `ID`에 접근하는 로직을 별도로 작성해야 하는 문제가 생길 수도 있습니다.

이처럼 컨벤션은 의사소통 비용과 작업 비용을 효율적으로 사용하기 위해서라면 정하는 게 정말 좋습니다. 개인 프로젝트를 할 때에도 과거의 자신과 소통한다고 생각하시면, 과거에 작성한 코드의 의미도 빠르게 파악할 수 있습니다. 그렇다면, 컨벤션을 정확히 어디에서 정하는 게 좋을까요?

**- 컨벤션을 정해야 하는 곳**

컨벤션을 아주 세세한 것까지 정한다고 가정해 봅시다. 어디에서 세미콜론을 쓸지까지 한 줄 한 줄 정하는 겁니다. 다른 동료들은 줄이 끝날 때마다 세미콜론을 적지만, 프론트엔드 개발자의 경우 세미콜론이 자동 삽입되므로 여태까지 적지 않은 사람도 있을 겁니다. 그렇다면 함수 하나를 선언할 때도 세미콜론을 쓸지 말지 정하고 있어야 합니다. 디자이너의 경우 임시로 만든 사각형 레이어의 이름까지 컨벤션을 찾아가면서 바꿔야 합니다. 이렇게 자세하게 컨벤션을 정하는 또 다른 문제점은, 서로를 가장 잘 알고 있어야 한다는 것입니다. 프로젝트를 위해 임시로 모인 사람들의 경우 어떻게 서로의 습관을 알 수 있을까요? 물론 대화를 많이 해본다면 되지만, 당장 프로젝트의 마감 기한이 3주 뒤라면 이런 저런 대화를 하면서 컨벤션을 정할 시간을 할애할 수 있을까요? 그럴 수 없을 가능성이 매우 큽니다. 

프로젝트를 하다보면 어디에서 컨벤션을 정하고 시작하면 좋을 지 알 수 있습니다. 여기에서 중요한 점은 프로젝트를 '시작'하는 것입니다. 컨벤션은 추후에도 바뀔 수 있고, 새로 생길 수 있습니다. 그러니 컨벤션을 일부만 정하고 세부적인 내용은 필요할 때마다 정하는 걸 추천드립니다.

프로젝트 초기 세팅 시, 프론트엔드와 백엔드 개발자는 일관성 있는 코드 작성과 협업을 위해 특정한 컨벤션을 정의해야 합니다. 아래는 각 영역에서 고려할 수 있는 일반적인 컨벤션과 가이드라인입니다.

프론트엔드 (Frontend) 컨벤션:

- 폴더 구조:프로젝트의 폴더 구조를 명확히 정의하고, 컴포넌트, 스타일, 리소스 등의 디렉토리를 일관성 있게 구성합니다.
- 컴포넌트 명명 규칙:컴포넌트의 이름을 명확하게 부여하고, 적절한 명명 규칙을 따르도록 합니다. 예를 들어, CamelCase 또는 PascalCase 등을 사용할 수 있습니다.
- 상태 관리:상태 관리 라이브러리 (예: Redux) 사용 여부와 관련된 가이드라인을 설정합니다.
컴포넌트 간 상태 전파 및 관리에 대한 일관성 있는 접근 방식을 정의합니다.
- 코딩 스타일:코드 작성 시에 들여쓰기, 세미콜론 사용, 따옴표 사용 등의 코딩 스타일을 통일하고, ESLint나 Prettier 등의 도구를 사용하여 일관성 있는 코드 스타일을 유지합니다.
- API 통신:API 호출 방식과 에러 핸들링 방법을 정의하고, API 엔드포인트의 명명 규칙을 일관성 있게 유지합니다.

백엔드 (Backend) 컨벤션:

- 프로젝트 구조:프로젝트의 디렉토리 구조를 명확하게 정의하고, 모듈, 미들웨어, 라우터 등을 일관성 있게 배치합니다.
- 코딩 스타일:일관된 들여쓰기, 네이밍 규칙, 코드 스타일 등을 정의하고, 필요한 경우 Linter를 도입하여 코드 품질을 유지합니다.
- API 설계 및 문서화:RESTful 또는 GraphQL과 같은 API 설계 원칙을 명시하고, API 문서를 작성하여 프론트엔드와 협업을 용이하게 합니다.
- 데이터베이스 관리:데이터베이스 스키마와 모델의 명명 규칙을 정의하고, 마이그레이션과 시딩에 대한 일관성 있는 접근 방식을 설정합니다.
- 보안:인증과 권한 부여 방식을 명시하고, 보안 취약성에 대한 방어책을 마련합니다.
- 로깅과 모니터링:로그 작성 방식과 모니터링 도구의 사용에 대한 가이드라인을 설정하여 시스템의 상태를 추적하고 개선합니다.

이러한 컨벤션과 가이드라인은 팀 간 협업을 원활하게 하고, 코드의 일관성과 품질을 유지하는 데 도움이 됩니다. 프로젝트 특성에 따라 이러한 규칙을 조정하거나 추가할 필요가 있습니다.

디자이너:

- 디자인 시스템 구축: 디자이너는 일관된 디자인을 유지하기 위해 디자인 시스템을 구축하거나 기존 시스템을 업데이트할 수 있습니다.
컬러 팔레트, 폰트, 아이콘 등을 정의하고 개발자와 협력하여 시각적 일관성을 확보합니다.
- 컴포넌트 및 스타일 가이드 작성:디자이너는 프론트엔드 개발자와 협업하여 UI 컴포넌트 및 스타일 가이드를 작성합니다.
이를 통해 프론트엔드 개발자는 일관된 디자인을 구현할 수 있습니다.
- 프로토타입 제작:프로젝트 초기에는 사용자 경험과 디자인을 시연하기 위한 프로토타입을 제작합니다.
프로토타입은 디자인 결정에 대한 팀 간 의사소통을 촉진하고 사용자 피드백을 수렴하는데 사용됩니다.

프로젝트 매니저 (PM):

- 일정 및 리소스 관리:PM은 프로젝트 일정을 세우고, 개발자, 디자이너, 품질 담당자 등의 리소스를 관리합니다.
초기에는 개발 시작일과 마감일을 설정하고, 필요한 인력 및 장비를 확보합니다.
- 프로젝트 범위 및 목표 정의:프로젝트 목표와 범위를 정의하고, 개발자 및 디자이너와 협력하여 프로젝트의 주요 기능과 우선 순위를 결정합니다.
팀 간 커뮤니케이션 촉진:팀 간 원활한 소통을 도모하고, 디자이너, 개발자, 품질 담당자 간의 협업을 촉진합니다.
주기적인 미팅을 조직하고 이슈 및 진행 상황을 모니터링합니다.
- 위험 관리:프로젝트 초기에는 가능한 위험을 식별하고 관리 계획을 수립합니다.
팀원들과 협력하여 예상치 못한 문제에 대비하고 조치를 취합니다.
- 자원 할당 및 역할 분배:디자이너와 개발자에게 역할과 책임을 할당하고, 팀원들 간의 협업을 최적화합니다.
필요한 경우 추가 자원을 적절히 할당합니다.
- 프로젝트 문서화:프로젝트 요구사항, 일정, 리소스 계획 등을 문서화하여 팀원들 간에 명확한 이해를 도모합니다.
변경 사항이나 프로젝트 진행 상황을 문서로 기록하여 추적 가능하게 합니다.

다양한 역할들이 프로젝트에서 함께 작업하면서 필요한 문서 종류가 있습니다. 아래는 각 역할별로 작성해야 하는 주요 문서 종류를 나열한 것입니다.

개발자 (Frontend 및 Backend):
- 요구사항 명세서 (Requirements Specification):프로젝트의 목표, 기능, 사용자 요구사항 등을 상세히 기술한 문서입니다.
- 기술 설계서 (Technical Design Document):소프트웨어 시스템의 아키텍처, 데이터베이스 설계, API 명세 등 기술적인 측면을 다루는 문서입니다.
- 코드 문서화 (Code Documentation):코드의 동작 방식, 함수 및 클래스의 역할, API 사용법 등을 설명하는 문서입니다.
- 테스트 계획서 (Test Plan):테스트의 목적, 전략, 테스트 케이스, 테스트 일정 등을 정의하는 문서입니다.
- 버전 관리 로그 (Version Control Log):코드의 변경 내용, 버전 업데이트 로그를 기록하는 문서입니다.

디자이너:
- 디자인 가이드 (Design Guidelines):디자인 시스템, 컬러 팔레트, 폰트, 아이콘 사용 등의 가이드라인을 포함한 문서입니다.
- UI/UX 디자인 문서 (UI/UX Design Document):사용자 인터페이스 및 경험에 대한 설계, 와이어프레임, 프로토타입 등을 다루는 문서입니다.
- 디자인 리서치 보고서 (Design Research Report):사용자 연구 및 조사 결과를 정리한 문서로, 디자인 결정의 근거를 제공합니다.
- 그래픽 자산 명세서 (Graphic Assets Specification):그래픽 자산의 종류, 크기, 해상도 등을 명세하는 문서입니다.

프로젝트 매니저:
- 프로젝트 계획서 (Project Plan):프로젝트 일정, 리소스 할당, 범위, 위험 관리 계획 등이 포함된 문서입니다.
- 회의록 (Meeting Minutes):회의에서 토의된 내용, 결정 사항, 할당된 역할 등을 기록하는 문서입니다.
- 위험 관리 계획서 (Risk Management Plan):프로젝트에서 예상되는 위험과 대응 전략을 정의하는 문서입니다.
- 프로젝트 이슈 트래킹 (Project Issue Tracking):발생한 이슈, 버그, 변경 요청 등을 기록하고 추적하는 문서입니다.
- 프로젝트 보고서 (Project Report):주기적으로 프로젝트의 진행 상황, 비용, 위험 등을 요약하는 문서입니다.

이러한 문서들은 프로젝트의 효율성과 투명성을 유지하는 데 기여하며, 팀 간의 원활한 협업을 도모합니다. 각 문서의 구성과 내용은 프로젝트의 특성에 따라 조정될 수 있습니다.
